# 用户身份识别方案

## 问题说明

当用户通过 REST API 提交反馈到 GitHub Discussions 时，如何确认提交人的身份信息？

## GitHub Discussions API 的特性

根据 GitHub Discussions API 的机制：

1. **作者身份由 Token 决定**
    - 创建讨论时，GitHub 会将调用 API 时使用的 Token 对应的用户记录为讨论的作者
    - 如果使用服务器的统一 Token，所有讨论都会显示为该 Token 对应的用户创建的

2. **无法在创建时指定其他用户为作者**
    - GitHub API 不支持在创建讨论时指定其他用户作为作者
    - `author` 字段是只读的，只能通过查询获取，不能设置

## 解决方案

### 方案1：在反馈内容中包含用户信息（推荐用于 MVP）

**实现方式：**

- 客户端在提交反馈时，包含用户信息字段（用户名、邮箱、GitHub 用户名等）
- 服务器端使用统一的 Token 创建讨论
- 在讨论的 `body` 中包含用户信息作为元数据

**优点：**

- ✅ 实现简单，快速上线
- ✅ 不需要用户授权
- ✅ 用户体验好，无需额外操作

**缺点：**

- ❌ 无法验证用户身份的真实性
- ❌ 用户可能提供虚假信息
- ❌ Discussions 中显示的作者是服务器账号，不是实际用户

**实现示例：**

```java
// REST API 请求体
{
  "title": "功能建议：支持批量生成",
  "content": "希望添加批量生成 Javadoc 的功能",
  "userInfo": {
    "name": "张三",
    "email": "zhangsan@example.com",
    "githubUsername": "zhangsan",
    "pluginVersion": "1.0.0",
    "ideaVersion": "2023.3"
  }
}

// 生成到 Discussions 的 body
String body = """
    ## 反馈内容

    {content}

    ---

    ## 提交人信息

    - **姓名**: {userName}
    - **邮箱**: {userEmail}
    - **GitHub**: @{githubUsername}
    - **插件版本**: {pluginVersion}
    - **IDEA 版本**: {ideaVersion}
    - **提交时间**: {timestamp}
    """;
```

### 方案2：GitHub OAuth 认证（推荐用于生产环境）

**实现方式：**

- 用户在插件中通过 GitHub OAuth 登录
- 获取用户的 GitHub Access Token
- 使用用户的 Token 创建讨论，讨论会显示为用户本人创建
- 或者在用户信息中包含 GitHub Token，服务器使用用户的 Token 创建

**优点：**

- ✅ 真实可靠，讨论显示为实际用户创建
- ✅ 可以在 Discussions 中直接 @ 提及用户
- ✅ 用户身份可验证

**缺点：**

- ❌ 实现复杂，需要 OAuth 流程
- ❌ 用户需要授权，可能增加使用门槛
- ❌ 需要处理 Token 的存储和安全问题

**实现流程：**

1. 插件端：用户点击"登录 GitHub"按钮
2. 插件端：跳转到 GitHub OAuth 授权页面
3. GitHub：用户授权后返回授权码
4. 插件端：将授权码发送到服务器
5. 服务器端：使用授权码交换 Access Token
6. 服务器端：存储用户的 Token（加密存储）
7. 提交反馈时：使用用户的 Token 创建讨论

### 方案3：GitHub 用户名 + @ 提及（混合方案）

**实现方式：**

- 客户端在提交反馈时，包含用户的 GitHub 用户名（可选）
- 服务器端使用统一的 Token 创建讨论
- 在讨论内容中使用 GitHub @ 提及功能（`@username`）
- GitHub 会自动通知被 @ 的用户

**优点：**

- ✅ 实现简单
- ✅ 用户可以收到通知
- ✅ 不强制要求用户登录

**缺点：**

- ❌ 无法验证用户名是否真实
- ❌ 讨论作者仍然是服务器账号
- ❌ @ 提及需要用户在仓库中有权限

**实现示例：**

```java
String body = """
    ## 反馈内容

    {content}

    ---

    **提交人**: @{githubUsername}

    {userInfo}
    """;
```

### 方案4：匿名反馈 + 可选的 GitHub 信息

**实现方式：**

- 支持完全匿名提交
- 可选地收集 GitHub 用户名，用于后续联系
- 使用统一的服务器账号创建讨论
- 在讨论中标记为"匿名反馈"或"来自插件用户"

**优点：**

- ✅ 最简化的用户体验
- ✅ 保护用户隐私
- ✅ 实现最简单

**缺点：**

- ❌ 无法追溯反馈来源
- ❌ 难以与用户进一步沟通

## 推荐方案

### 阶段1：MVP（最小可行产品）

**采用方案1 + 方案3 的混合：**

- 在反馈内容中包含用户信息（姓名、邮箱、GitHub 用户名）
- 如果提供了 GitHub 用户名，在内容中使用 @ 提及
- 使用服务器统一 Token 创建讨论
- 在讨论标题或内容中标记为"插件反馈"

**反馈内容模板：**

```markdown
## 反馈内容

{用户提交的反馈内容}

---

## 提交人信息

- **姓名**: {name}
- **邮箱**: {email}
- **GitHub**: @{githubUsername}（如果提供）
- **插件版本**: {pluginVersion}
- **IDEA 版本**: {ideaVersion}
- **提交时间**: {timestamp}
- **反馈类型**: {type}（Bug/功能建议/使用问题）
```

### 阶段2：生产环境（可选）

如果反馈量增加，需要更好的用户管理，可以考虑：

- 实现方案2（GitHub OAuth）
- 或者要求用户在提交反馈时提供 GitHub 用户名，并验证其真实性

## API 设计建议

### REST API 请求格式

```json
{
  "title": "反馈标题",
  "content": "反馈详细内容",
  "type": "bug|feature|question",
  "category": "general|ideas|qa",
  "userInfo": {
    "name": "用户姓名（可选）",
    "email": "用户邮箱（可选）",
    "githubUsername": "GitHub 用户名（可选）",
    "pluginVersion": "插件版本",
    "ideaVersion": "IDEA 版本",
    "os": "操作系统信息"
  },
  "metadata": {
    "clientId": "客户端唯一标识",
    "timestamp": "提交时间戳"
  }
}
```

### 响应格式

```json
{
  "success": true,
  "discussion": {
    "id": "讨论 ID",
    "number": "讨论编号",
    "url": "讨论链接",
    "title": "讨论标题"
  },
  "message": "反馈已成功提交"
}
```

## 安全考虑

1. **输入验证**
    - 验证用户输入的格式（邮箱格式、GitHub 用户名格式等）
    - 防止 XSS 攻击（对用户输入进行转义）
    - 限制内容长度

2. **速率限制**
    - 对同一客户端 IP 或 clientId 进行速率限制
    - 防止恶意提交大量反馈

3. **敏感信息处理**
    - 不要记录用户的密码或敏感信息
    - 邮箱地址可以部分隐藏（如 `zh***@example.com`）

4. **Token 安全**
    - 如果使用用户的 Token，必须加密存储
    - 定期轮换服务器 Token
    - 使用环境变量存储 Token，不要提交到代码库
